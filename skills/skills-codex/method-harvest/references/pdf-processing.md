# PDF Processing Contract

Use this contract after `fulltext-acquire` has landed an identity-matched PDF and before extracting a method card. The source PDF is immutable evidence; OCR text, page images, and receipts are derived artifacts.

## 1. Deterministic Preflight

Run the local inspector and provide at least one distinctive title token. Multiple tokens are conjunctive.

```bash
python3 "$METHOD_HARVEST_SKILL_DIR/scripts/inspect_pdf.py" PAPER.pdf \
  --work-id graham_harvey_popadak_rajgopal_2017 \
  --artifact-id corporate-culture-field-w23255-open \
  --parent-artifact-id not_applicable \
  --artifact-role main_paper \
  --version-identity "NBER Working Paper 23255, March 2017 PDF" \
  --doi-or-source-id "10.3386/w23255" \
  --expect-title-token "distinctive title phrase" \
  --expect-author-token "lead author surname" \
  --expect-doi "10.xxxx/example" \
  --render-page 14 \
  --render-page 46 \
  --render-required-pages \
  --render-output-dir method-harvest/pdf-processing/rendered \
  --output method-harvest/pdf-processing/PAPER_ID_PDF_PROCESSING.json
```

Identity has two levels. `work_id` is the stable METHOD_CARD/synthesis ID; `artifact_id` is the exact `FULLTEXT_MANIFEST` row for this acquired version. A main PDF uses `parent_artifact_id=not_applicable`. A separate appendix, questionnaire, codebook, or supplement has its own `artifact_id` and points `parent_artifact_id` to the main PDF's artifact ID. The inspector also emits `paper_id=work_id` and `parent_paper_id=parent_artifact_id` as compatibility aliases; new consumers should use the explicit names. It rejects placeholders, invalid roles, self-parented companions, and main PDFs with a companion parent. `doi_or_source_id` binds the receipt to the DOI, repository number, or publisher record used in the manifest; when it contains a DOI, it must match `--expect-doi`.

The receipt remains `aris.method-harvest.pdf-inspection.v2` and adds these identity and render blocks without changing existing text-layer fields. An older v2 receipt that lacks `artifact_identity` or native `render_evidence` is readable continuity evidence, but it must be regenerated by the inspector before current acceptance; do not migrate it by manually inserting JSON fields.

The receipt must record:

- source path, byte size, SHA-256, PDF version, encryption status, and page count
- extraction tool/version and extracted-text hash
- per-page text-layer coverage, sparse/empty pages, and separate visual-review required/cleared/pending page lists
- `native_text`, `mixed_text_review`, or `ocr_required`
- identity evidence checked only on the first one or two viewer pages: title tokens plus optional author/DOI corroboration
- the work/artifact/version identity tuple used to join the manifest, METHOD_CARD, and synthesis
- inspector-generated `render_evidence` with source-PDF hash and actual page count; 1-based viewer page; repository-relative PNG path; PNG SHA-256, byte count, and independently parsed IHDR dimensions; and Poppler renderer/version
- immutable-source confirmation and required visual-check targets

`ok=true` means the PDF container and inspection succeeded. `ready_for_method_harvest=true` additionally requires a usable text layer, `identity.status=pass`, and no pending sparse-page review. `identity.status=not_checked` is never ready. Identity tokens are intentionally limited to the first pages so a target title appearing only in the reference list cannot validate the wrong PDF.

Use repeatable `--render-page N` for focal definitions, formulas, tables, or appendices and `--render-required-pages` for every sparse/empty page flagged by the inspector. PNG output and the source PDF must be inside `--repo-root` (normally auto-detected), so `source_pdf` and every rendered `png_path` are repository-relative. In a nested strict-sandbox workspace, pass the outer repository root explicitly; argument parsing no longer probes for `.git` when `--repo-root` is supplied. The inspector stages each Poppler render, parses its PNG IHDR, hashes it, refuses to overwrite different existing bytes, and confirms the source PDF hash did not change. Do not hand-add or hand-edit `render_evidence` after the run.

When the inspector lists `visual_review_pending_pages`, open and inspect the generated PNGs, then rerun the same render command with one `--visual-review-cleared-page N` argument per genuinely reviewed page. A clear is accepted only when the same inspector run renders that page, so the final receipt cannot claim a sparse-page review without native PNG evidence. The inspector rejects out-of-range pages and pages it did not flag. A cleared page means it was visually reviewed; it does not assert that OCR is accurate or that the page is substantively irrelevant.

## 2. Text, OCR, And Page Mapping

- `native_text`: use page-preserving extraction such as `pdftotext -layout`. Sparse pages remain in `visual_review_required_pages` even when the overall usable-page ratio qualifies as native text.
- `mixed_text_review`: inspect every listed sparse page. OCR only material pages whose content is not recoverable from the text layer.
- `ocr_required`: stop method extraction. Run authorized **local** OCR, preserve the original PDF and original hash, and inspect the OCR derivative again. Never upload a licensed PDF to an unrelated OCR service.
- Keep viewer pages 1-based. OCR must not change the citation map silently; if a derivative inserts or removes pages, maintain an explicit original-viewer-page → derivative-page map.
- Keep each rendered filename's `-pN` token aligned with its 1-based viewer page. The inspector names native PNG evidence from `artifact_id`, not `work_id`, so separately acquired versions cannot collide silently.
- Do not overwrite or “repair” the source PDF. Record both source and derivative hashes.

OCR is a transcription aid, not ground truth. Compare names, minus signs, decimal points, superscripts, subscripts, formula operators, and table alignment against rendered pages.

## 3. Two-Pass Reading

### Pass A — Structure map

Locate the title/version, abstract, theory and hypotheses, institutional setting, sample construction, variable definitions, empirical specification, main results, robustness, limitations, references, and material appendices/questionnaires. Record viewer-page ranges before drafting claims.

### Pass B — Evidence extraction

Read the actual paragraphs, equations, table notes, and appendix definitions that support each required method-card cell. Text search finds candidates; it does not replace reading the surrounding page.

Always render and visually inspect pages containing:

- focal construct and variable definitions
- sample-flow counts or exclusion rules
- the main equation/specification
- main and null-result tables used in the review
- factor loadings, index construction, survey items, or questionnaires
- any text/table contradiction or OCR-sensitive numeric claim

## 4. Evidence Location Rules

- Cite `PDF p.<viewer-page>` as the canonical location.
- Add `printed p.<page>` or `article p.<page>` only when useful; never substitute it for viewer pages.
- A material claim needs a paragraph, equation, table, figure, or appendix pointer.
- Store short evidence notes or tight paraphrases, not an unbounded copy of licensed fulltext.
- Mark unreadable or contradictory evidence `needs_verification`; do not infer a conventional formula or repair a number.

## Acceptance

- Current processing receipt matches the source PDF hash and page count.
- Receipt identity joins the stable work ID to the exact manifest artifact/version/source ID; companion parentage is explicit and non-self-referential.
- Identity is verified and the text-layer/OCR route is explicit.
- Source PDF remains unchanged; derivatives have separate hashes.
- Critical equations, tables, and appendices have inspector-generated, hash- and page-bound PNG evidence; `render_evidence` was not injected after inspection.
- Every material method-card claim retains a viewer-page location.
