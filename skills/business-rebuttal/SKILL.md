# Business Rebuttal

Review package: $ARGUMENTS

## Purpose

Turn referee reports into a grounded response plan and revision letter for business-school journals.

## Inputs

Need:

- manuscript or paper directory
- referee reports or editor letter
- target journal and round
- current tables and results
- user-approved new analyses or commitments

## Workflow

### Phase 1: Normalize Reviews

Create issue cards with:

- reviewer or editor ID
- raw concern anchor
- issue type
- severity
- required evidence
- response status

Issue types:

- identification
- construct validity
- measurement
- data and sample
- alternative explanation
- omitted variable
- timing or pre-trend
- inference and clustering
- robustness
- economic magnitude
- mechanism
- literature positioning
- writing clarity
- journal fit

### Phase 2: Build Revision Strategy

Group issues into:

- can answer with current text
- needs re-analysis already feasible
- needs new data or manual decision
- must concede or narrow claim
- outside feasible scope

### Phase 3: Draft Responses

Use this response shape:

1. acknowledge the concern
2. state the action taken or planned
3. cite exact manuscript location or table
4. explain why the change addresses the concern
5. preserve caveats where evidence remains partial

### Phase 4: Commitment Gate

Every promise must map to:

- already implemented
- user-approved revision
- planned but still pending
- explicitly deferred with reason

### Phase 5: Write Artifacts

When writing is allowed, create:

- `rebuttal/ISSUE_BOARD.md`
- `rebuttal/REVISION_PLAN.md`
- `rebuttal/RESPONSE_LETTER_DRAFT.md`
- `rebuttal/EDITOR_SUMMARY.md`

## Rules

- Match every factual claim to a manuscript location, table, or user-confirmed result.
- Narrow claims when the evidence cannot answer the referee's causal concern.
- Keep tone professional and specific.
- Track editor-level requests separately from referee requests.
- Preserve unresolved items in the revision plan.
