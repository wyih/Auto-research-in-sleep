> **v0.4.19** (2026-06-14) вҖ” **honesty / guardrails patch** (theme from a Codex fresh-eyes audit; no behavior change for healthy setups). **рҹ”ҙ MCP protocol-version negotiation guard** вҖ” the stdio handshake requested `2025-03-26` but never read the version the server negotiated back (a parsed-but-dead field), so a server agreeing on a version ARIS can't speak was silently accepted and later `tools/list` / `tools/call` ran on an incompatible protocol with opaque failures. ARIS now validates the negotiated version against a supported set (`2025-11-25` / `2025-06-18` / `2025-03-26` / `2024-11-05` вҖ” stdio framing is identical across these) and, on an unsupported one, terminates the child + clears the slot + surfaces a clear per-server error (`aris doctor` shows it) вҖ” the "terminate when versions can't be agreed" behavior the MCP lifecycle spec requires. The *request* stays `2025-03-26` (proven against `codex mcp-server`), so healthy servers are unaffected вҖ” verified end-to-end: the real Codex MCP server still spawns + initializes + advertises its tools under the guard. **рҹ§№ Papercuts**: the OpenAI-family subagent fail-loud message dropped its stale "lands in v0.4.18" marker (now version-agnostic + actionable, still credential-free); OpenAI upstream error bodies are now truncated (500 chars) + credential-redacted (`sk-вҖҰ` keys and `Bearer вҖҰ` tokens, via a substring scanner that catches the compact-JSON shape `{"api_key":"sk-вҖҰ"}` a misconfigured proxy can reflect back) instead of splatted verbatim; the system-prompt hook-events summary now counts only the hooks the runtime actually executes (a `command` hook with a `command` string), matching the parser. Tests (CI mode): api 32 / runtime 204 / tools 67 / aris-cli 167 / commands 5, all green; live smoke confirms the real Codex MCP server still initializes under the guard. Reviewed by Codex MCP (gpt-5.5 xhigh): design GO вҶ’ impl NO-GO (compact-secret miss + command-string strictness) вҶ’ GO after fixes.

> **v0.4.18** (2026-06-14) вҖ” **default model вҶ’ Claude Opus 4.8**, with corrected pricing and a safety net. It fires only on that precise 404 (never 400/rate-limit/auth), latches against loops, and the text path rebuilds from a pre-turn snapshot so a retry never double-appends the user message; accounts **with** 4.8 are byte-identical to a plain bump. **рҹ’° Pricing corrected** (verified against Anthropic's published schedule; had been a 3вҖ“5Г— over-estimate): current **Opus 4.5вҖ“4.8 = `$5/$25`** (deprecated Opus 4/4.1 keep `$15/$75`, split by word-boundary so a future `opus-4-10` isn't mis-tiered); **Sonnet 4.x = `$3/$15`** (decoupled from the generic unknown-model fallback, which stays `$15/$75`); Haiku was already correct. **рҹ§№ Backlog**: `aris setup` option 10 pins the Codex MCP reviewer to `model_reasoning_effort="xhigh"` (deterministic for new setups, independent of `~/.codex/config.toml`; idempotent merge never clobbers an existing entry); a startup + `aris doctor` **misconfig hint** ([#259](https://github.com/wanshuiyin/Auto-claude-code-research-in-sleep/issues/259)) for a silently-ignored/misplaced config (malformed JSON, or a stray `~/.aris/config.yaml` вҖ” stderr-only so `--print`/JSON stdout stays clean); the system-prompt hook summary now marks parsed-but-never-fired events **"PARSED ONLY вҖҰ will NOT run"** instead of implying dead hooks run (full event expansion вҖ” actually firing `SessionStart`/`SessionEnd`/вҖҰ вҖ” deferred to a separate issue). > **v0.4.15** (2026-05-29) вҖ” OpenAI-compatible streaming robustness hotfix. Closes [#249](https://github.com/wanshuiyin/Auto-claude-code-research-in-sleep/issues/249): MiniMax (and other OpenAI-compatible providers / proxies) were effectively unusable because the clean-EOF completion check treated the `data: [DONE]` SSE sentinel as the *only* authoritative signal. A non-empty `choices[].finish_reason` is the Chat Completions spec's terminal-chunk marker; `[DONE]` is a transport convention some compatible providers never emit (MiniMax sends `finish_reason: "stop"` then closes without `[DONE]`). The clean-EOF decision is now a pure, unit-tested `stream_eof_action(...)` that completes on EITHER `[DONE]` OR a non-empty `finish_reason`; reads are NOT stopped early at finish_reason (a trailing `include_usage` usage-only chunk is still consumed), genuine truncation still hard-errors, and a pre-output proxy abort still restarts. Coupled fixes: **OE7** reads finish_reason before the `delta` guard (delta-less terminal choice); **OE2** flushes pending tool calls on any non-empty finish_reason; **OE4** surfaces a mid-stream error envelope as a hard error instead of silently dropping it; **OE3** tolerates `data:{...}` without the space after the colon. +5 unit tests (77вҶ’82) extract the previously-untested SSE completion logic into pure helpers. Anthropic SSE path untouched. Codex MCP (gpt-5.5 xhigh) 3 rounds (GO-WITH-NITS вҶ’ GO-WITH-NITS вҶ’ GO).

> **v0.4.14** (2026-05-25) вҖ” Security-hygiene release closing the top items from the v0.4.13 codex audit (gpt-5.5 xhigh, 6/10 NEEDS-REWORK verdict). **рҹ”ҙ S9 (P0) system-prompt config redaction** вҖ” before v0.4.14, `render_config_section()` dumped the merged `settings.json` verbatim into the system prompt sent to the LLM provider, leaking `env`, `mcpServers.<name>.headers.Authorization` Bearer tokens, hook command env, signed-URL query params, and `apiKey` fields. Regression test exercises 9 distinct leak surfaces; URL parser has its own targeted test for 7 smuggling attempts including port-position secret injection (codex round-3 catch). **рҹҹЎ P9 (P1)**: DeepSeek `aris --help` now points at `aris setup` option 7 instead of an env-var path the resolver never honored. **рҹҹЎ M1/M2 (P1) doc**: `aris doctor` + README/README_CN gain experimental warning whenever `mcpServers.len() > 0` (full MCP tool dispatch lands v0.4.16). **рҹҹў C11 (P2) stream idle timeout** вҖ” both Anthropic `MessageStream` and the OpenAI SSE loop wrap `response.chunk().await` in `tokio::time::timeout` (env `ARIS_STREAM_IDLE_TIMEOUT_SECS`, default 120, clamp `[10, 1800]`, 0/negative disables); closes the "aris hangs forever with no output" symptom when an upstream HTTPS proxy holds a connection without keepalives. **Bundle**: 77 skills (+1 `/wiki-enrich` via late same-day sync to main `7e3ab67` which also picks up upstream `check_ready.sh` awk + grep-c null-match fix), 54 helpers. Custom [Claude Code](https://docs.anthropic.com/en/docs/claude-code) skills for autonomous ML research workflows. These skills orchestrate **cross-model collaboration** вҖ” Claude Code drives the research while an external LLM (via [Codex MCP](https://github.com/openai/codex)) acts as a critical reviewer. рҹ”Җ **Also supports [alternative model combinations](#alternative-model-combinations) (Kimi, LongCat, DeepSeek, etc.) вҖ” no Claude or OpenAI API required.** For example, [MiniMax-M3 + GLM-5 or GLM-5 + MiniMax-M3](docs/MiniMax-GLM-Configuration.md). рҹӨ– **[Codex CLI native](skills/skills-codex/)** вҖ” full skill set also available for OpenAI Codex. рҹ–ұпёҸ **[Cursor](docs/CURSOR_ADAPTATION.md)** вҖ” works in Cursor too. рҹ–ҘпёҸ **[Trae](docs/TRAE_ARIS_RUNBOOK_EN.md)** вҖ” ByteDance AI IDE. рҹҡҖ **[Antigravity](docs/ANTIGRAVITY_ADAPTATION.md)** вҖ” Google's agent-first IDE. рҹҗҷ **[Copilot CLI](docs/COPILOT_CLI_ADAPTATION.md)** вҖ” GitHub's terminal agent (native SKILL.md + MCP). рҹҶ“ **[Free tier via ModelScope](docs/MODELSCOPE_GUIDE.md) вҖ” zero cost, zero lock-in.**

> рҹ’ӯ **Why not self-play with a single model?** Using Claude Code subagents or agent teams for both execution and review is technically possible, but tends to fall into **local minima** вҖ” the same model reviewing its own patterns creates blind spots.
>
> *Think of it like adversarial vs. stochastic bandits: a single model self-reviewing is the stochastic case (predictable reward noise), while cross-model review is adversarial (the reviewer actively probes weaknesses the executor didn't anticipate) вҖ” and adversarial bandits are fundamentally harder to game.*
>
> рҹ’ӯ **Why two models, not more?** Two is the minimum needed to break self-play blind spots, and 2-player games converge to Nash equilibrium far more efficiently than n-player ones. Adding more reviewers increases API cost and coordination overhead with diminishing returns вҖ” the biggest gain is going from 1вҶ’2, not 2вҶ’4.
>
> Claude Code's strength is fast, fluid execution; Codex (GPT-5.5 xhigh) is slower but more deliberate and rigorous in critique. These complementary styles вҖ” **speed Г— rigor** вҖ” produce better outcomes than either model talking to itself.
>
> рҹ§ҝ **Want the strongest possible reviewer?** Add `вҖ” reviewer: oracle-pro` to any skill to route reviews through **GPT-5.5 Pro** via [Oracle MCP](https://github.com/steipete/oracle). Pro-level reasoning for proof verification, experiment auditing, and final stress tests. Works with API key or free browser mode. [Setup вҶ’](#-optional-gpt-54-pro-via-oracle)

## 1. рҹҺҜ More Than Just a Prompt

> These are full pipelines вҖ” you can also use each workflow independently. Already have an idea? Skip to Workflow 1.5. Have results? Jump to Workflow 3. Got reviews? Jump to Workflow 4. Want persistent memory? Enable [Research Wiki](#-research-wiki--persistent-research-memory). See [Quick Start](#quick-start) for all commands and [Workflows](#workflows) for the full breakdown.

**Basic mode** вҖ” give ARIS a research direction, it handles everything:

```
/research-pipeline "factorized gap in discrete diffusion LMs"
```

**рҹ”Ҙ Targeted mode** вҖ” got a paper you want to improve? Give ARIS the paper + the code:

```
/research-pipeline "improve method X" вҖ” ref paper: https://arxiv.org/abs/2406.04329, base repo: https://github.com/org/project
```

ARIS reads the paper вҶ’ finds its weaknesses вҶ’ clones the codebase вҶ’ generates ideas that specifically fix *those* weaknesses with *that* code вҶ’ runs experiments вҶ’ writes your paper. Like telling a research assistant: *"read this paper, use this repo, find what's missing, and fix it."*

> Mix and match: `ref paper` only = "what can be improved?", `base repo` only = "what can I build with this code?", both = "improve *this* paper using *this* code."

**рҹ”Ҙ Rebuttal mode** вҖ” reviews just dropped? Don't panic. ARIS reads every concern, builds a strategy, and drafts a rebuttal that's grounded, structured, and under the character limit:

```
/rebuttal "paper/ + reviews" вҖ” venue: ICML, character limit: 5000
```

Three safety gates вҖ” rebuttal will NOT finalize if any fails:
- рҹ”’ **No fabrication** вҖ” every claim maps to paper/review/user-confirmed result
- рҹ”’ **No overpromise** вҖ” every promise is user-approved
- рҹ”’ **Full coverage** вҖ” every reviewer concern is tracked

Two outputs: `PASTE_READY.txt` (exact char count, paste to venue) + `REBUTTAL_DRAFT_rich.md` (extended version for manual editing).

## 3. рҹҡҖ Quick Start

```bash

git clone https://github.com/wanshuiyin/Auto-claude-code-research-in-sleep.git
bash Auto-claude-code-research-in-sleep/tools/install_aris.sh ~/your-project   # symlinks ARIS skills into <project>/.claude/skills/
# (prefer a global install instead? cp -r Auto-claude-code-research-in-sleep/skills/* ~/.claude/skills/)

# 1b. Update later (when upstream changes)
cd Auto-claude-code-research-in-sleep && git pull
bash tools/smart_update.sh --apply   # updates safe skills, flags your personal customizations

# Optional Codex mirror managed project install
bash tools/install_aris_codex.sh ~/your-codex-project

# Managed Codex project update
cd Auto-claude-code-research-in-sleep && git pull
bash tools/install_aris_codex.sh ~/your-codex-project --reconcile

# Copied Codex installs only (not for projects installed by install_aris_codex.sh)
bash tools/smart_update_codex.sh --local ~/.codex/skills
bash tools/smart_update_codex.sh --local ~/.codex/skills --apply

# 2. Set up Codex MCP (for review skills)
npm install -g @openai/codex
codex setup                    # set model to gpt-5.5 when prompted
claude mcp add codex -s user -- codex mcp-server

# 3. Use in Claude Code
claude
> /idea-discovery "your research direction"  # Workflow 1 вҖ” be specific! not "NLP" but "factorized gap in discrete diffusion LMs"
> /experiment-bridge                         # Workflow 1.5 вҖ” have a plan? implement + deploy + collect results
> /auto-review-loop "your paper topic or scope"  # Workflow 2: review вҶ’ fix вҶ’ re-review overnight
> /paper-writing "NARRATIVE_REPORT.md"       # Workflow 3: narrative вҶ’ polished PDF

> /meta-optimize                                # Meta: analyze usage logs вҶ’ propose skill improvements
```

## 4. вңЁ Features

ARIS chains **79 composable skills** across the whole research lifecycle вҖ” literature & novelty вҶ’ idea discovery вҶ’ GPU experiments вҶ’ autonomous review loop вҶ’ paper writing вҶ’ peer review вҖ” with **cross-model adversarial review** (Claude executes В· GPT-5.5 xhigh reviews В· optional **GPT-5.5 Pro** via Oracle), anti-hallucination DBLP/CrossRef citations, a persistent **Research Wiki**, flexible model backends, human-in-the-loop checkpoints, and optional Feishu / Zotero / Obsidian / GPU integrations.

рҹ”Ҙ *And it scales to any agent's **ultracode-style deep mode** вҖ” the breadth/firepower pass adapts to the runtime (Claude Code ultracode + workflows on Opus 4.8, Codex `spawn_agent`, or plain sequential), feeding three roles: **breadth В· cross-model review вҶ’ accuracy В· research wiki вҶ’ memory**. However a loop is driven, it reports to the same cross-model jury + research wiki вҖ” **it can drive, never acquit**.*

## 8. рҹ”„ Workflows

These skills compose into a full research lifecycle. Each workflow can be used independently or chained together:

- **Exploring a new area (e.g., writing a survey)?** Start with Workflow 1 вҶ’ `/idea-discovery`
- **Have a plan, need to implement and run?** Workflow 1.5 вҶ’ `/experiment-bridge`
- **Already have results, need iterative improvement?** Workflow 2 вҶ’ `/auto-review-loop`
- **Ready to write the paper?** Workflow 3 вҶ’ `/paper-writing` (or step by step: `/paper-plan` вҶ’ `/paper-figure` вҶ’ `/paper-write` вҶ’ `/paper-compile` вҶ’ `/auto-paper-improvement-loop`)
- **Got reviews back? Need to rebuttal?** Workflow 4 вҶ’ `/rebuttal` вҖ” parse reviews, draft safe rebuttal, follow-up rounds
- **Full pipeline?** Workflow 1 вҶ’ 1.5 вҶ’ 2 вҶ’ 3 вҶ’ submit вҶ’ 4 вҶ’ `/research-pipeline` + `/rebuttal` вҖ” from idea through submission and rebuttal
- **Want ARIS to remember and learn?** рҹ“ҡ `/research-wiki init` вҖ” persistent memory across sessions. Papers, ideas, failed experiments compound over time
- **Want ARIS to improve itself?** Workflow M вҶ’ `/meta-optimize` вҖ” analyze usage logs, propose skill improvements, reviewer-gated

> вҡ пёҸ **Important:** These tools accelerate research, but they don't replace your own critical thinking. Always review generated ideas with your domain expertise, question the assumptions, and make the final call yourself. The best research comes from human insight + AI execution, not full autopilot.

### Workflow 1: Idea Discovery & Method Refinement рҹ”Қ

> **"What's the state of the art? Where are the gaps? How do we solve it?"**

Don't have a concrete idea yet? Just give a research direction вҖ” `/idea-discovery` handles the rest:

1. рҹ“ҡ **Survey** the landscape (recent papers, open problems, recurring limitations)
2. рҹ§  **Brainstorm** 8-12 concrete ideas via GPT-5.5 xhigh
3. рҹ”Қ **Filter** by feasibility, compute cost, and quick novelty search
4. рҹӣЎпёҸ **Validate** top ideas with deep novelty check + devil's advocate review
5. рҹ§Ә **Pilot** top 2-3 ideas in parallel on different GPUs (30 min - 2 hr each)
6. рҹҸҶ **Rank** by empirical signal вҖ” ideas with positive pilot results rise to the top
7. рҹ”¬ **Refine** the top idea into a problem-anchored proposal via iterative GPT-5.5 review
8. рҹ§Ә **Plan** claim-driven experiments with ablations, budgets, and run order

The output is a ranked `IDEA_REPORT.md` plus a refined proposal (`refine-logs/FINAL_PROPOSAL.md`) and experiment plan (`refine-logs/EXPERIMENT_PLAN.md`) for the top idea. Dead-end ideas are documented too, saving future exploration.

### Workflow 2: Auto Research Loop рҹ”Ғ (sleep & wake up to results)

> **"Review my paper, fix what's wrong, repeat until it's good."**
>
> GPT-5.5 reviews вҶ’ identifies weaknesses вҶ’ suggests experiments вҶ’ Claude Code writes scripts, deploys to GPU, monitors results, rewrites the paper вҖ” all while you sleep. Just add your [GPU server config](#gpu-server-setup) to `CLAUDE.md`.

1. рҹ”Қ **Deep review** вҖ” GPT-5.5 xhigh reviews the current paper / claims / experiments and identifies weaknesses
2. рҹ©№ **Fix** вҖ” Claude implements the fixes (rewrites sections, adds baselines, or runs new experiments via `/run-experiment`); skips any experiment estimated > 4 GPU-hours and flags it for manual follow-up
3. рҹ“Ҡ **Re-evaluate** вҖ” collect results via `/monitor-experiment`, update paper, feed back to the reviewer
4. рҹ”Ғ **Repeat** вҖ” until score вүҘ `POSITIVE_THRESHOLD` (default 6/10) or `MAX_ROUNDS` (default 4) is hit; if context window fills mid-loop, the workflow auto-resumes from `REVIEW_STATE.json`

<a id="-research-wiki--persistent-research-memory"></a>

### рҹ“ҡ Research Wiki вҖ” Persistent Research Memory

> **"Stop re-deriving. Start compounding."** вҖ” inspired by [Karpathy's LLM Wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)

Without the wiki, ARIS is stateless вҖ” every `/idea-discovery` starts from scratch. With the wiki, ARIS accumulates knowledge across the entire research lifecycle: papers read, ideas tested, experiments run, claims verified or invalidated.

**The key insight:** failed ideas are the most valuable memory. A researcher who knows what doesn't work generates better ideas than one starting from zero.

**Setup:**
```
> /research-wiki init     # one-time, creates research-wiki/ in your project
```

**That's it.** Once initialized, the wiki works automatically.

<details>
<summary><b>Show the automatic wiki hooks</b> вҖ” what fires at /research-lit, /idea-creator, /result-to-claim, plus the re-ideation nudge</summary>

| When | What happens | Wiki action |
|------|-------------|-------------|
| `/research-lit` finds papers | Papers auto-ingested | `papers/<slug>.md` created, edges added, query_pack rebuilt |
| `/idea-creator` runs | Reads wiki first | Failed ideas = banlist, gaps = search seeds, papers = known prior work |
| `/idea-creator` finishes | ALL ideas written back | Both recommended AND eliminated ideas вҶ’ `ideas/<id>.md` |
| `/result-to-claim` judges | Results written back | Experiment page created, claim status updated (supported/invalidated) |
| 3+ ideas fail | Re-ideation suggested | "рҹ’Ў Consider re-running /idea-creator вҖ” the wiki now knows what doesn't work" |

</details>

<details>
<summary><b>Show Research Wiki data model</b> вҖ” Paper / Idea / Experiment / Claim entities and the typed graph edges that connect them</summary>

**Four entity types:**

| Entity | What it stores | Example |
|--------|---------------|---------|
| рҹ“„ **Paper** | Structured summary: thesis, method, limitations, reusable ingredients | `paper:chen2025_factorized_gap` |
| рҹ’Ў **Idea** | Hypothesis, status (proposed/failed/succeeded), failure notes, lessons | `idea:001` |
| рҹ§Ә **Experiment** | Metrics, verdict, hardware, duration | `exp:001` |
| рҹ“Ӣ **Claim** | Testable statement + evidence status (reported/supported/invalidated) | `claim:C1` |

**Typed relationships** (stored in `graph/edges.jsonl`):
```
paper --extends--> paper              idea --inspired_by--> paper
paper --contradicts--> paper          idea --tested_by--> experiment
paper --addresses_gap--> gap          experiment --supports--> claim
paper --supersedes--> paper           experiment --invalidates--> claim
```

</details>

<details>
<summary><b>Show Research Wiki spiral-learning example and manual subcommands</b> вҖ” failed ideas вҶ’ better ideas across 3 rounds; ingest / query / update / lint / stats</summary>

**Spiral learning in action:**
```
Round 1: read 15 papers вҶ’ wiki remembers вҶ’ idea A вҶ’ experiment вҶ’ FAIL
         wiki records: "A fails because OOM at batch>32, loss diverges"

Round 2: /idea-creator reads wiki вҶ’ sees A failed вҶ’ generates idea D (avoids A's trap)
         вҶ’ experiment вҶ’ PARTIAL SUCCESS
         wiki records: "D works on small models, fails on large"

Round 3: /idea-creator reads wiki вҶ’ knows A failed + D partial вҶ’ generates idea F
         (combines D's success with new approach) вҶ’ experiment вҶ’ SUCCESS рҹҺү
```

**Subcommands:**
```
/research-wiki init                              # initialize wiki
/research-wiki ingest "paper title" вҖ” arxiv: xxx  # manually add a paper
/research-wiki query "topic"                      # rebuild query_pack.md
/research-wiki update idea:001 вҖ” outcome: negative # update entity
/research-wiki lint                               # health check (orphans, contradictions, stale claims)
/research-wiki stats                              # overview (paper/idea/experiment/claim counts)
```

</details>

> рҹ”’ **Safe by design:** All workflow hooks are guarded by `if research-wiki/ exists`. No wiki = no impact. Zero dependencies (pure Python stdlib). You choose when to enable it.

### Workflow M: Meta-Optimize рҹ§¬ (ARIS optimizes itself)

> **"Analyze my usage patterns and improve your own skills."**

Unlike Workflows 1вҖ“4 which optimize *research artifacts* (papers, code, experiments), Workflow M optimizes the *harness itself* вҖ” the SKILL.md instructions, default parameters, and convergence rules that govern how ARIS operates. Inspired by [Meta-Harness](https://arxiv.org/abs/2603.28052) (Lee et al., 2026).

<details>
<summary><b>Show Workflow M one-time setup and usage commands</b> вҖ” Claude Code hook install, /meta-optimize variants (project / per-skill / --global / apply)</summary>

**Setup (one-time, in normal terminal):**
```bash
mkdir -p .claude .aris/meta tools/meta_opt
cp Auto-claude-code-research-in-sleep/templates/claude-hooks/meta_logging.json .claude/settings.json
cp Auto-claude-code-research-in-sleep/tools/meta_opt/*.sh tools/meta_opt/
chmod +x tools/meta_opt/*.sh
claude   # hooks active immediately
```

**Usage (after 5+ workflow runs):**
```
> /meta-optimize                        # analyze current project
> /meta-optimize "auto-review-loop"     # focus on one skill
> /meta-optimize --global               # analyze trends across ALL projects
> /meta-optimize apply 1                # apply recommended change #1
```

</details>

**How it works:**

1. рҹ“Ҡ **Passive logging** вҖ” Claude Code hooks silently record every skill invocation, tool call, failure, parameter override, and user prompt. Events are written to **both** project-level (`.aris/meta/events.jsonl`) and global (`~/.aris/meta/events.jsonl`, with a `"project"` tag) logs. Zero user effort.
2. рҹ”Қ **Pattern analysis** вҖ” `/meta-optimize` reads the log and identifies:
   - Parameters users override most often (bad defaults)
   - Tools that fail repeatedly in specific skills (missing error handling)
   - Review score plateaus (convergence rules too loose/tight)
   - Manual corrections users make (skill gaps)
3. рҹ©№ **Patch proposal** вҖ” generates minimal diffs to target SKILL.md files with data-backed justifications
4. рҹ”¬ **Reviewer gate** вҖ” GPT-5.5 xhigh reviews each patch: does the evidence support it? could it hurt other users?
5. вң… **User approval** вҖ” only applied with explicit user consent. All changes are logged and reversible.

### 10.2 Install Skills

> рҹ’Ў **Recommended: project-local flat symlink install** (since 2026-04-20). Each ARIS skill is symlinked individually into `.claude/skills/<skill-name>`, so Claude Code's slash-command discovery picks them up. A manifest at `.aris/installed-skills.txt` tracks what ARIS installed вҖ” uninstall and reconcile only ever touch managed entries, never your own skills.
>
> рҹӨ– **Codex mirror route:** keep Claude on `install_aris.sh` / `smart_update.sh`. For Codex-native project installs, use `install_aris_codex.sh`; for copied Codex installs, use `smart_update_codex.sh`.

```bash
# 1. Clone ARIS once to a stable location
git clone https://github.com/wanshuiyin/Auto-claude-code-research-in-sleep.git ~/aris_repo

# 2. For each project that uses ARIS, attach via symlinks:
cd ~/your-paper-project
bash ~/aris_repo/tools/install_aris.sh
# вҶ’ creates one symlink per skill: .claude/skills/<skill> вҶ’ ~/aris_repo/skills/<skill>
# вҶ’ writes manifest .aris/installed-skills.txt (tracks every entry ARIS installed)
# вҶ’ updates managed CLAUDE.md ARIS block (best-effort, compare-and-swap)
# вҶ’ re-runnable: rerun anytime to reconcile new/removed upstream skills

# 3. To update existing skills' content for ALL attached projects:
cd ~/aris_repo && git pull   # symlinks resolve to live upstream вҖ” content updates automatically

# 3a. To pick up newly added or removed upstream skills, rerun the installer:
bash ~/aris_repo/tools/install_aris.sh ~/your-paper-project   # adds new symlinks, removes broken ones

# Other useful flags:
bash ~/aris_repo/tools/install_aris.sh --dry-run        # show plan, no changes
bash ~/aris_repo/tools/install_aris.sh --uninstall      # remove only managed symlinks (per manifest)
bash ~/aris_repo/tools/install_aris.sh --from-old       # migrate from old nested .claude/skills/aris/

# Windows (PowerShell, no WSL required; creates flat per-skill junctions):
.\tools\install_aris.ps1 C:\path\to\your-paper-project -Platform claude
.\tools\install_aris.ps1 C:\path\to\your-codex-project -Platform codex
```

**Why "git pull" alone isn't enough for new/removed skills:** the flat layout uses one symlink per skill, so upstream additions/deletions don't propagate until the installer is re-run. The trade-off bought us Claude Code's automatic slash-command discovery (which only scans one directory level deep).
